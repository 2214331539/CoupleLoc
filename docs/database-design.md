# CoupleLoc Database Design

This document is the living source of truth for the backend database schema.
Whenever `server/app/models.py` or an Alembic migration changes, update this
file in the same change set.

Current migrations:

- `server/alembic/versions/0001_initial.py`
- `server/alembic/versions/0002_shared_features.py`
- `server/alembic/versions/0003_sharing_privacy_options.py`

## Design Goals

- Store only the latest location for each user in the MVP.
- Do not store historical tracks.
- Allow exactly one active partner pair per user.
- Keep location visibility controlled by explicit sharing settings.
- Keep the schema simple enough for a two-person private deployment, while
  leaving room for device records, push tokens, and geofences later.

## Entity Overview

| Table | Purpose |
| --- | --- |
| `users` | Login identity and display profile. |
| `pairing_invites` | One-time invite codes used to pair two users. |
| `couples` | Active two-person relationship records. |
| `sharing_settings` | Per-user location sharing switch, mode, and privacy preferences. |
| `latest_locations` | One overwritten location row per user. |
| `chat_messages` | Text and quick-status messages between the pair. |
| `calendar_events` | Shared plans, visits, anniversaries, and reminders. |
| `memory_points` | Shared map pins saved as relationship memories. |

## `users`

Stores the minimal account data needed for private use.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key, generated by the API. |
| `username` | `varchar(64)` | yes | Unique login name, normalized to lowercase. |
| `display_name` | `varchar(64)` | yes | User-facing name shown in the app. |
| `password_hash` | `varchar(255)` | yes | PBKDF2-HMAC-SHA256 hash. Never store raw passwords. |
| `created_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Unique index: `ix_users_username` on `username`

Delete behavior:

- Deleting a user cascades to invites, sharing settings, and latest location.
- Current schema does not yet cascade `couples` from both sides through ORM;
  database foreign keys cascade from `couples.user_a_id` and `couples.user_b_id`.

## `pairing_invites`

Stores temporary invite codes for pairing.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key. |
| `code` | `varchar(16)` | yes | Unique invite code, currently 6 uppercase letters/digits. |
| `creator_user_id` | `uuid` | yes | User who created the invite. |
| `expires_at` | `timestamptz` | yes | Invite expires after 24 hours. |
| `consumed_at` | `timestamptz` | no | Set when another user accepts the invite. |
| `created_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Unique index: `ix_pairing_invites_code` on `code`
- Index: `ix_pairing_invites_expires_at` on `expires_at`
- Foreign key: `creator_user_id -> users.id on delete cascade`

Lifecycle:

- New invite: `consumed_at = null`.
- Accepted invite: `consumed_at` set to the acceptance timestamp.
- Expired invites are rejected by the API. A cleanup job can delete old rows later.

## `couples`

Stores a pair of users that can view each other's live location when sharing is
enabled.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key. |
| `user_a_id` | `uuid` | yes | One side of the pair. |
| `user_b_id` | `uuid` | yes | Other side of the pair. |
| `status` | `varchar(16)` | yes | Currently `active`; future values may include `ended`. |
| `created_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Unique constraint: `uq_couple_pair` on `(user_a_id, user_b_id)`
- Index: `ix_couples_status` on `status`
- Foreign keys:
  - `user_a_id -> users.id on delete cascade`
  - `user_b_id -> users.id on delete cascade`

Important invariant:

- The API currently checks that a user has no existing active couple before
  creating a new pair.
- The database does not yet enforce "only one active couple per user" with a
  partial unique index. Add one if this becomes a public multi-user system.

## `sharing_settings`

Stores the user's explicit location sharing switch, sharing mode, temporary
expiration, and privacy preferences.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `user_id` | `uuid` | yes | Primary key and foreign key to `users.id`. |
| `enabled` | `boolean` | yes | `true` means the API accepts location uploads and partner reads. |
| `mode` | `varchar(24)` | yes | `always`, `one_hour`, `foreground`, or `paused`. |
| `expires_at` | `timestamptz` | no | Temporary sharing expiration, currently used by `one_hour`. |
| `share_battery` | `boolean` | yes | Whether partner responses include battery level and charging state. |
| `share_distance` | `boolean` | yes | Whether the frontend should show pair distance. |
| `precise_location` | `boolean` | yes | Whether partner responses include exact coordinates. |
| `updated_at` | `timestamptz` | yes | Last time the setting changed. |

Indexes and constraints:

- Primary key: `user_id`
- Foreign key: `user_id -> users.id on delete cascade`

Behavior:

- A new account gets `enabled = true`, `mode = always`, and all privacy shares enabled.
- `PATCH /locations/sharing` updates this row. Sending `mode = one_hour` sets
  `expires_at` to one hour from the server clock unless a specific expiration is
  supplied. Sending `mode = paused` disables sharing.
- When this setting changes, the API sends a `sharing.updated` WebSocket event
  to the partner if connected.
- When `enabled = false`, the API rejects new uploads from that user and blocks
  partner reads from `GET /locations/partner/latest`.
- When `mode = foreground`, the API rejects background-source location uploads.
- When `precise_location = false`, partner-facing location responses are rounded
  and their accuracy is raised to at least 500 meters. The owner still sees exact
  coordinates.
- When `share_battery = false`, partner-facing location responses omit
  `battery_level` and `is_charging`.

Privacy note:

- Disabling sharing does not delete the previous `latest_locations` row. It
  only makes it inaccessible to the partner. Add explicit deletion if the
  product later requires "pause and erase last known location".

## `latest_locations`

Stores only the most recent location per user.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `user_id` | `uuid` | yes | Primary key and foreign key to `users.id`. |
| `latitude` | `double precision` | yes | WGS84 latitude, validated by the API. |
| `longitude` | `double precision` | yes | WGS84 longitude, validated by the API. |
| `accuracy` | `double precision` | no | Meter-level accuracy from the device when available. |
| `speed` | `double precision` | no | Device-reported speed when available. |
| `heading` | `double precision` | no | Device-reported heading when available. |
| `battery_level` | `double precision` | no | 0 to 1 battery level. |
| `is_charging` | `boolean` | no | Whether the device is charging. |
| `source` | `varchar(16)` | yes | `foreground` or `background`. |
| `recorded_at` | `timestamptz` | yes | Device location timestamp. |
| `received_at` | `timestamptz` | yes | Server receipt/update timestamp. |

Indexes and constraints:

- Primary key: `user_id`
- Foreign key: `user_id -> users.id on delete cascade`

Write behavior:

- `POST /locations/me` upserts this row by user ID.
- No historical rows are inserted.
- The API broadcasts a `location.updated` WebSocket event to the partner after
  a successful update.

Retention:

- Current retention is one row per user.
- There is no `location_history` table in the MVP.

Low battery behavior:

- The API compares `battery_level` with `LOW_BATTERY_THRESHOLD`.
- When a user crosses from above the threshold to at or below the threshold,
  the API sends a realtime `battery.low` event to the partner.
- Low battery alerts are not persisted in the database in the current design.

## `chat_messages`

Stores lightweight pair chat. This is not intended to be a full chat platform;
it exists for quick relationship status updates and short text messages.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key. |
| `couple_id` | `uuid` | yes | Pair that owns the message. |
| `sender_user_id` | `uuid` | yes | User who sent the message. |
| `message_type` | `varchar(24)` | yes | `text` or `quick_status`. |
| `body` | `varchar(500)` | yes | Message text shown in the app. |
| `status_key` | `varchar(64)` | no | Stable key for quick statuses. |
| `created_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Index: `ix_chat_messages_created_at` on `created_at`
- Foreign keys:
  - `couple_id -> couples.id on delete cascade`
  - `sender_user_id -> users.id on delete cascade`

Behavior:

- `GET /chat/messages` returns recent messages for the active couple.
- `POST /chat/messages` creates a message and sends a realtime
  `chat.message_created` event to the partner.
- Quick status messages must include `status_key`.

Retention:

- Chat retention is currently indefinite.
- Add retention or export/delete controls before public use.

## `calendar_events`

Stores shared calendar items.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key. |
| `couple_id` | `uuid` | yes | Pair that owns the event. |
| `created_by_user_id` | `uuid` | yes | User who created the event. |
| `title` | `varchar(120)` | yes | Event title. |
| `notes` | `text` | no | Optional event notes. |
| `starts_at` | `timestamptz` | yes | Event start time. |
| `ends_at` | `timestamptz` | no | Optional event end time. |
| `created_at` | `timestamptz` | yes | Server timestamp. |
| `updated_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Index: `ix_calendar_events_starts_at` on `starts_at`
- Foreign keys:
  - `couple_id -> couples.id on delete cascade`
  - `created_by_user_id -> users.id on delete cascade`

Behavior:

- `GET /calendar/events` lists all events for the active couple.
- `POST /calendar/events` creates an event.
- `PATCH /calendar/events/{event_id}` updates an event.
- `DELETE /calendar/events/{event_id}` deletes an event.
- Creates, updates, and deletes send `calendar.event_changed` to the partner.

Validation:

- `ends_at` must be absent or later than `starts_at`.

## `memory_points`

Stores shared map memory pins.

| Column | Type | Required | Notes |
| --- | --- | --- | --- |
| `id` | `uuid` | yes | Primary key. |
| `couple_id` | `uuid` | yes | Pair that owns the point. |
| `created_by_user_id` | `uuid` | yes | User who created the point. |
| `title` | `varchar(120)` | yes | Memory title. |
| `note` | `text` | no | Optional memory note. |
| `latitude` | `double precision` | yes | WGS84 latitude. |
| `longitude` | `double precision` | yes | WGS84 longitude. |
| `created_at` | `timestamptz` | yes | Server timestamp. |
| `updated_at` | `timestamptz` | yes | Server timestamp. |

Indexes and constraints:

- Primary key: `id`
- Foreign keys:
  - `couple_id -> couples.id on delete cascade`
  - `created_by_user_id -> users.id on delete cascade`

Behavior:

- `GET /memories/points` lists memory points for the active couple.
- `POST /memories/points` creates a point.
- `DELETE /memories/points/{point_id}` deletes a point.
- Creates and deletes send `memory.point_changed` to the partner.

Current frontend behavior:

- The mobile app saves a memory point from the user's latest uploaded location.
- The map renders memory points as purple markers.

## Realtime Events

Realtime events are not persisted in the database.

`location.updated`:

```json
{
  "type": "location.updated",
  "location": {
    "user_id": "uuid",
    "latitude": 31.2304,
    "longitude": 121.4737,
    "accuracy": 12,
    "speed": null,
    "heading": null,
    "battery_level": 0.82,
    "is_charging": false,
    "source": "foreground",
    "recorded_at": "2026-05-02T12:00:00Z",
    "received_at": "2026-05-02T12:00:01Z"
  }
}
```

```json

  "type": "sharing.updated",
```

## Future Schema Candidates

These are intentionally not part of the current implementation:

- `devices`: device name, platform, last seen timestamp.
- `push_tokens`: Expo push token per device.
- `location_history`: optional short retention history.
- `geofences`: arrival alerts for home/school/work.
- `audit_logs`: explicit sharing and login audit trail.
- `notification_events`: persisted push or alert delivery attempts.

When adding any of these, create a new Alembic revision and update this
document in the same patch.
